## Cursor Hackathon: Build Eric's Software Factory

## Event Overview

This version of the event is tighter, more credible, and easier to judge.

Instead of a broad collection of tweet-inspired ideas, the hackathon will focus on a smaller set of high-signal tracks grounded in Eric Zakariasson's recent posts and validated by Cursor's current product direction, especially automations, cloud agents, Bugbot Autofix, skills, subagents, and long-running agents.

The goal is not "build random things from Twitter." The goal is to prototype the next generation of always-on software workflows: agents that trigger themselves, verify their own work, and improve real systems over time.

## Event Details

- **Venue:** Dawn Capital HQ
- **Target size:** 100 participants
- **Event type:** In-person Cursor hackathon
- **Recommended team size:** 1-4 people
- **Recommended build window:** 3 hours
- **Demo length:** 3-5 minutes max per team
- **Event style:** Short, high-energy, demo-first

## Core Positioning

**Build Eric's Software Factory**

This hackathon is inspired by Eric Zakariasson's recent posts and Cursor's latest product direction. Participants will build durable product and workflow ideas, not one-off gimmicks, with a focus on automations, review loops, agent tooling, and self-improving software systems.

## The Four Tracks

### 1. Always-On Agents

Build automations that trigger from real-world events or schedules and produce useful outcomes without manual prompting during the run.

Examples:
- PR or issue triggered agents
- Slack or webhook driven agents
- incident response agents
- scheduled repo maintenance or reporting agents

What good looks like:
- a real trigger
- a real action
- a verifiable output such as a PR, report, fix, or comment

### 2. Review + QA

Build systems that review, test, debug, or verify agent work before humans have to step in.

Examples:
- Bugbot-style autofix loops
- visual regression agents
- multi-browser QA with subagents
- evidence-driven debug systems
- traceability and audit tools for generated code

What good looks like:
- catches something real
- explains why it matters
- proves the fix or diagnosis with evidence

### 3. Agent Runtime Tools

Build the infrastructure that makes agents more capable, composable, and useful in real workflows.

Examples:
- agent skills libraries
- model routing systems
- package or tool selection engines
- MCP-based workflows across business systems
- reusable capability layers for coding agents

What good looks like:
- improves agent quality or speed
- can be reused across tasks
- feels like a real primitive, not a one-off hack

### 4. Software Factory

Build systems where many agents, tools, and feedback loops work together to continuously improve software over time.

Examples:
- multi-agent coordination systems
- self-improving codebase loops
- background planning plus implementation pipelines
- agent fleets with judging, retries, and verification

What good looks like:
- multiple steps or workers
- clear control of drift, conflict, or failures
- believable path toward repeated operation, not a one-shot demo

## Recommended Event Flow

Based on previous Cursor hackathon learnings, keep the structure short and disciplined.

### Pre-Event

- Release the four tracks and 8-12 example prompts in advance
- Allow optional team formation beforehand
- Ask registrants about Cursor experience level
- Prepare setup instructions ahead of time
- Confirm WiFi, power, demo screen, and backup AV setup
- Decide how credits will be distributed before the event

### Event Day Schedule

#### 9:00 - Arrival and Setup

- check-in
- breakfast / coffee
- team formation
- setup support

#### 9:45 - Opening Remarks

- welcome
- explain the four tracks
- explain judging criteria
- explain side challenges
- explain submission rules

#### 10:00 - Examples and Q&A

- walk through a few strong example ideas
- explain what a good demo looks like
- clarify rules on AI usage and pre-built code

#### 10:15 - Hacking Starts

- teams commit to one main track
- mentors and organizers circulate

#### 13:15 - Hacking Ends

- hard stop
- collect submission links
- teams prep demos

#### 13:30 - Demo Block

- 3-5 minutes per team
- strict timer
- no exceptions

#### 15:00 - Judging and Awards

- shortlist if needed
- judges deliberate
- announce track winners and side challenge winners

## Judging Criteria

Use one shared rubric for all tracks.

- **Concrete Workflow Value:** Does this solve a real developer or team problem?
- **Track Fit:** Is the project clearly aligned with one of the four tracks?
- **Reliability and Verification:** Does it show evidence, testing, checks, or auditability?
- **Technical Execution:** Is the implementation believable and well put together?
- **Demo Clarity:** Can the team explain the workflow crisply in a few minutes?

## Rules

- Teams of 1-4
- Public GitHub repo required
- Working demo required
- AI-assisted development encouraged
- Open-source libraries allowed
- Pre-event planning allowed
- Core implementation should be produced during the event

If you want to discourage recycled demos, use a lightweight integrity review on submitted repos before finalist selection.

## Operations Guidance

### Venue and Logistics

- Strong WiFi is mandatory
- Make sure there are enough outlets
- Have more than one organizer running the event
- Bring name tags
- Use a dedicated presentation network if possible

### Credits

- Do not use open redemption links
- Give every participant their own credit allocation, even for team submissions
- Require individual registration
- Choose one distribution method in advance:
  - CSV emailer
  - redemption portal
  - QR distribution at check-in

### Team Formation

- Team matchmaking helps a lot
- Individual registration prevents team captain issues
- Maximum team size should stay at 4

### Presentations

- AV test before demos
- Use a visible timer
- Keep demos short and consistent
